# Infrastructure Catalogue: ai-swarm-infra

## 1. Functional Definition
The ai-swarm-infra tool is a distributed agent orchestration scaffold designed for local area networks. It implements a multi-node topology where a central Python orchestrator manages specialised LLM workers via standard HTTP calls to remote Ollama instances. Unlike centralised inference setups, this architecture maps specific hardware profiles to distinct personas: a high-performance Ryzen 5 node serves as the Coder (running qwen2.5-coder:7b) while a legacy Intel MacBook Pro functions as the Reviewer (running llama3.2:3b). The system uses a linear task pipeline where the orchestrator decomposes a prompt, dispatches it to the Coder for raw logic, and subsequently routes the output to the Reviewer for optimisation and documentation.

## 2. The Problem of Framework Bloat
Prince initiated this project to bypass the hardware constraints of individual machines and the excessive complexity of contemporary agentic frameworks. Standard libraries such as CrewAI and AutoGen provide high-level abstractions that frequently obscure underlying HTTP mechanics and introduce significant dependency overhead. Furthermore, running multiple high-parameter models on a single machine often leads to VRAM exhaustion and thermal throttling. By building a custom Python-native client, Prince can exploit heterogeneous hardware (the home "intranet cluster") to parallelise the development lifecycle. This approach provides a granular mechanical understanding of agentic workflows while ensuring that the orchestrator remains lightweight and decoupled from the heavy lifting performed by the worker nodes.

## 3. Position in the 2026 Landscape
By April 2026, the local LLM ecosystem has bifurcated into model-layer and agent-layer tooling. While primitives such as llama.cpp RPC and Exo have matured to allow the sharding of trillion-parameter models across networked GPUs, they focus entirely on distributed inference. The ai-swarm-infra tool sits one level higher in the stack as an agent-layer manager. It is not a model server, nor does it attempt peer-to-peer weight splitting like the Exo alpha. Instead, it treats every machine as a discrete "whole-model" worker. It sits on top of inference clusters (such as K2 or olol) to orchestrate complex, multi-step reasoning chains. It is a workflow coordinator that assumes the underlying models are already served and ready for instruction.

## 4. Integration with BorAI Ops
Within the BorAI ecosystem, this tool is categorised under ops/ rather than apps/. This classification reflects its status as infrastructure shared across multiple applications and lifecycle tooling rather than a standalone product. At present, ai-swarm-infra exists as a seed within the ops/ directory: a structural claim to a future capability rather than an active component of the current stack. Once matured, it will provide a concrete capability for distributed agent orchestration that allows any BorAI application to delegate tasks to a network of local models. The use of Python 3.11 and uv-management ensures that the infrastructure remains portable across Linux and macOS environments without version conflicts.

## 5. Architectural Caveats
The current iteration of ai-swarm-infra is a structural scaffold and is not yet functionally runnable. The Python modules (main.py, orchestrator.py, and network_client.py) exist as skeletons to illustrate the intended logic flow. For this to transition into production-ready infrastructure, several components require implementation: the requests-based network client must be hardened against connection timeouts, and the orchestrator needs a robust state machine to handle worker failures. The hardware-to-IP mapping is currently hardcoded in the documentation. This must be moved to a dynamic configuration or environment-based discovery system before it can be established as active infrastructure within the BorAI vault.
