## 2. Frameworks: who exposes the spectrum, who smuggles in a default posture — with real primitives

### LangGraph — graph-native routing (explicit spectrum)

**Primitives:** `StateGraph`, `add_conditional_edges`, `Send` (dynamic fan-out), and `Command` (update state **and** route in one return). The graph API overview is [here](https://docs.langchain.com/oss/python/langgraph/graph-api).

**Conditional edges (task-level branch on state):**

```python
graph.add_conditional_edges("node_a", routing_function)
graph.add_conditional_edges("node_a", routing_function, {True: "node_b", False: "node_c"})
```

**`Command` (state + control flow — the “router returns patches” pattern):**

```python
from langgraph.types import Command

def my_node(state: State) -> Command[Literal["my_other_node"]]:
    return Command(
        update={"foo": "bar"},
        goto="my_other_node",
    )
```

**Trenches gotcha (often absent from tutorials):** the docs warn that **`Command` adds dynamic edges but static edges still run** — if you both return `Command(goto=...)` *and* `add_edge(node_a, node_b)`, you can get **parallel unintended execution**. That is a failure mode Copilot-style “paste a graph” rarely catches.

**Another trenches signal:** [Issue #5867 — LangGraph API server merges HTTP headers into runtime context](https://github.com/langchain-ai/langgraph/issues/5867) is the sort of production foot-gun that changes how you wire routing + auth: your **router** is only as safe as the runtime context schema.

**Default posture:** LangGraph *assumes* nothing about monolith vs swarm — but **most examples** collapse to “ReAct in one node” because that ships fastest.

### AutoGen — GroupChat routing (explicit, but ecosystem-split)

**0.2.x / AG2 lineage — `GroupChat` + `speaker_selection_method`:** strategies include `"auto"` (LLM selector), `"round_robin"`, `"random"`, `"manual"`, or a **callable** `(last_speaker, groupchat) -> Agent | str | None`. FSM-style constraints appear via `allowed_or_disallowed_speaker_transitions` (see Microsoft’s [FSM GroupChat article](https://microsoft.github.io/autogen/0.2/blog/2024/02/11/FSM-GroupChat/) and the [customised speaker selection notebook](https://microsoft.github.io/autogen/0.2/docs/notebooks/agentchat_groupchat_customized/)).

**Trenches fork:** AutoGen has **split trajectories** — Microsoft’s `microsoft/autogen` **0.4 rewrite** vs **AG2** continuing **0.2.x** ergonomics. Reddit and migration threads (e.g. [r/AutoGenAI discussion](https://www.reddit.com/r/AutoGenAI/comments/1gvusph/whats_going_on_with_autogen_and_ag2/)) are closer to ground truth than any single README. **Copilot often hallucinates which package’s `GroupChat` you mean.**

**Routing pain in the wild:** [Issue #2472 — custom speaker selection + tool calling](https://github.com/microsoft/autogen/issues/2472) captures a recurring trenches complaint: **orchestration features interact badly with tool registration**, so your “clean router” becomes a debugging maze.

**Feature pressure:** [Issue #3215 — speaker_selection_method should allow subset of agents](https://github.com/microsoft/autogen/issues/3215) shows users hitting the ceiling of expressiveness for **task-level routing** without writing bespoke selectors.

### CrewAI — process-level default + hierarchical “manager as router”

**Primitives:** `Process.sequential` (fixed pipeline; output feeds forward) vs `Process.hierarchical` (manager delegates). Docs: [Processes](https://docs.crewai.com/en/concepts/processes), [Hierarchical process](https://docs.crewai.com/en/learn/hierarchical-process).

```python
from crewai import Crew, Process

crew = Crew(
    agents=[agent1, agent2],
    tasks=[task1, task2],
    process=Process.hierarchical,
    manager_llm="gpt-4o",
)
```

**Explicit spectrum?** Partially. Sequential is **architecture as constant**; hierarchical is **architecture as runtime policy** (manager decides). For DAG-like control, practitioners drift toward **Flows** (see community tutorials such as [CrewAI forum: Flows](https://community.crewai.com/t/new-tutorial-crewai-flows-made-easy/7123)).

**Trenches gotcha:** forum threads (e.g. [manager_agent / hierarchical confusion](https://github.com/crewAIInc/crewAI/discussions/1220), [only manager visible in `self.agents`](https://community.crewai.com/t/issue-only-manager-agent-appears-in-self-agents-when-using-process-hierarchical/7010)) show **delegation failing silently** — the “router” exists on paper but the manager executes everything. That looks like a granularity win in diagrams and a monolith in traces.

### Claude Agent SDK — subagents as **hard isolation** (spectrum via spawn, not shared memory)

Official pattern: register programmatic subagents on `ClaudeAgentOptions`, and allow the `Agent` tool. Documentation: [Subagents in the SDK](https://code.claude.com/docs/en/agent-sdk/subagents).

```python
from claude_agent_sdk import query, ClaudeAgentOptions, AgentDefinition

async for message in query(
    prompt="Review the authentication module for security issues",
    options=ClaudeAgentOptions(
        allowed_tools=["Read", "Grep", "Glob", "Agent"],
        agents={
            "security-reviewer": AgentDefinition(
                description="Identifies security vulnerabilities",
                prompt="You are a security specialist...",
                tools=["Read", "Grep", "Glob"],
                model="opus",
            )
        },
    ),
):
    ...
```

**Undocumented-in-spirit but stated in docs:** subagents run in **fresh, isolated** conversations; the parent typically receives **only the final result**. That is not a limitation — it *is* the routing primitive: **you are buying focus and parallelisability by paying information loss.**

Hooks (`SubagentStart` / `SubagentStop`) are where observability must land; otherwise you are blind.

### OpenAI Agents SDK — handoffs as **tools** + explicit transcript surgery

Handoffs are first-class and, crucially, **represented as tools** the model invokes (`transfer_to_*`). Docs: [Handoffs](https://openai.github.io/openai-agents-python/handoffs/).

```python
from agents import Agent, handoff

billing_agent = Agent(name="Billing agent")
refund_agent = Agent(name="Refund agent")

triage_agent = Agent(
    name="Triage agent",
    handoffs=[billing_agent, handoff(refund_agent)],
)
```

**Spectrum mechanism most Copilot summaries skip:** `handoff(..., input_filter=...)` and run-level `RunConfig.handoff_input_filter` / `nest_handoff_history` (beta) **reshape what the next agent sees**. There is even a packaged filter:

```python
from agents import Agent, handoff
from agents.extensions import handoff_filters

handoff_obj = handoff(
    agent=Agent(name="FAQ agent"),
    input_filter=handoff_filters.remove_all_tools,
)
```

**Trenches detail:** per-handoff `input_filter` **wins** over `RunConfig.handoff_input_filter` when both are set (documented on the same page). Nested handoff history is **opt-in and still stabilising** — behaviour your tests must pin.

### Pydantic AI — three layers of spectrum (tool delegation, programmatic hand-off, graph)

Official guide: [Multi-Agent Patterns](https://pydantic.dev/docs/ai/guides/multi-agent-applications/).

**Delegation inside a tool (parent resumes after child returns):**

```python
from pydantic_ai import Agent, RunContext, UsageLimits

joke_selection_agent = Agent(
  'openai:gpt-5.2',
  instructions=(
      'Use the `joke_factory` to generate some jokes, then choose the best. '
      'You must return just a single joke.'
  ),
)
joke_generation_agent = Agent(
  'google-gla:gemini-3-flash-preview', output_type=list[str]
)

@joke_selection_agent.tool
async def joke_factory(ctx: RunContext[None], count: int) -> list[str]:
  r = await joke_generation_agent.run(
      f'Please generate {count} jokes.',
      usage=ctx.usage,
  )
  return r.output

result = joke_selection_agent.run_sync(
  'Tell me a joke.',
  usage_limits=UsageLimits(request_limit=5, total_tokens_limit=500),
)
```

**Programmatic hand-off (application code owns the router):** same page demonstrates **successive agents** with shared `RunUsage` and different `message_history` handling — this is the cleanest “**task-level switch in Python, not in the model**” pattern.

**Graph-based control:** points to [Pydantic AI graph docs](https://pydantic.dev/docs/ai/graph/) for when tool-delegation DAGs become unreadable.

**Observability hook:** `logfire.instrument_pydantic_ai()` is documented as the way to see **which agent failed** and **token usage per delegate** — important because delegation composes *requests* multiplicatively.

## 3. Where context stripping helps vs hurts

### Where stripping **degrades** (literature-consistent)

- **Lost-in-the-middle / U-shaped attention:** long contexts do not yield uniform recall; middle positions suffer. This is the classical justification for **splitting work** so each call’s prompt is **short and structured**.
- **Context rot in agent traces:** tool outputs, errors, and retries accumulate; models attend to the wrong artefacts and repeat failed strategies. This is more operational than a single paper citation — but it is why production systems **summarise, checkpoint, or shard**.

### Where stripping **improves** (under-discussed in docs)

- **Tool-heavy JSON and APIs:** passing full HTTP payloads forward is anti-performance; **extract-then-pass** (structured extraction) reduces hallucinated field bindings. Pydantic AI’s entire design centre — validated outputs — is a form of **controlled stripping** to the *schema-relevant* slice.
- **Security and compliance:** subagents (Claude SDK) and handoff filters (OpenAI SDK) intentionally **prevent** the specialist from inheriting secrets or toxic transcript noise.
- **Forcing the model to stop “narrating” and start “doing”:** narrow briefs improve adherence in coding agents when the harness supplies **local file context** via tools instead of dumping the repo into the prompt.

**Trenches consensus:** stripping is a **performance knob** and a **safety knob**; it is only “bad” when you strip **the minimum sufficient state** for recovery — which is Q6.

## 4. Coordination-cost curve: does granularity dominate its own overhead?

**Qualitative shape (robust):** adding agents increases **fan-out communication**, **serialisation points**, and **failure modes** faster than it increases capability — unless tasks are embarrassingly parallel or strongly role-specialised.

**Quantitative anchors (treat as directional, not universal laws):**

- [*Stop Wasting Your Tokens: Towards Efficient Runtime Multi-Agent Systems* (arXiv:2510.26585)](https://arxiv.org/html/2510.26585v2) analyses a supervisory pattern and reports supervisor overhead on the order of **~15% of tokens** and meaningful **latency inflation** in their analysis (exact percentages vary by task; read the overhead section rather than trusting second-hand summaries).
- Practitioner heuristics (blog synthesis such as [Sparrow on coordination costs](https://blog.sparrow.so/scaling-llm-agents-beyond-demos-coordination-costs-topologies-and-what-actually-works/)) argue coordination tokens crowd out “thinking” tokens; treat these as **sanity checks**, not physics.

**Mechanisms that dominate past a threshold**

- **Token overhead:** every hop re-encodes goals, constraints, partial plans, and tool transcripts — often with **more** verbosity than a monolith would have used if it stayed silent.
- **Latency:** sequential handoffs stack model RTTs; parallel subagents shift load to **your** orchestrator and downstream rate limits.
- **Error compounding:** τ-bench’s **pass^k** ethos exists because **multi-step reliability is multiplicative**, not additive.

**Opinion:** there *is* a threshold, but it is **task- and model-dependent**; the only honest calibration is **your** trace + dollars + p(success).

## 5. Measurement surfaces: what makes the spectrum visible

What teams actually instrument when they are serious:

- **Token spend and cost attribution** — per agent / per handoff / per tool (Pydantic AI’s `ctx.usage` threading is a deliberate pattern; OpenAI SDK separates handoff history policies; LangSmith traces wrap LangGraph nodes).
- **Task completion and graded rubrics** — SWE-bench-style patch application; τ-bench-style DB state; GAIA-style outcome checks.
- **Drift / behaviour variance** — not always called “drift”; often **eval harnesses** (regression suites of prompts), **tool-call diff stats**, **replay tests**.
- **Human interventions** — approvals, edits, reroutes, escalations; in enterprise, this is sometimes the *only* ground truth that the model is misaligned.

**Platforms (representative, not exhaustive):**

- **LangSmith** — deep LangChain/LangGraph integration; node-level traces are the selling point.
- **Langfuse** — OSS-friendly tracing, experiments, and cost tracking; common in self-hosted stacks.
- **AgentOps** — session-oriented agent observability (competes in the same buyer bracket; check current feature matrix before committing).
- **OpenTelemetry GenAI** — vendor-neutral span semantics including agent operations (`invoke_agent`, tool execution naming). Spec: [GenAI agent spans](https://opentelemetry.io/docs/specs/semconv/gen-ai/gen-ai-agent-spans/) and the blog [AI Agent Observability (OpenTelemetry, 2025)](https://opentelemetry.io/blog/2025/ai-agent-observability/). **Stability is still evolving** — expect `OTEL_SEMCONV_STABILITY_OPT_IN` style flags.

**Trenches point:** Copilot reads README feature lists; production teams read **whether your router increments a span attribute** that lets you aggregate `p(success | agent=X)` — without that, architectural debates are astrology.

## 6. Blind execution and debugging: net failure cost

**Sub-agent / handoff failure without parent context**

- **Diagnosis cost rises:** the failing leaf often lacked the **global constraint** that made the failure inevitable (repo conventions, prior user preference, partial error only visible two hops ago).
- **Recovery cost is bifurcated:** if the harness returns **rich artefacts** (logs, tool traces, structured error), recovery can be fast; if it returns **only a summary**, you pay a human or a parent model to **reconstruct state** — sometimes more expensive than letting a monolith fail *with* the transcript.

**Monolith failure with full context**

- **Attribution cost rises:** the trace is long; root cause may be “turn 37 went wrong” — harder for humans, sometimes easier for automated replay.
- **Retry cost:** naïve retries re-burn tokens; summarisation introduces new failure modes.

**Net assessment (consensus-leaning):**

- Isolation **reduces** debug surface *per step* and **increases** integration-debug *between* steps.
- The economically rational pattern matches [2505.18286](https://arxiv.org/abs/2505.18286): **cascade** between granular and monolithic modes — route wide, execute narrow, escalate to wide context when metrics say you must.

## Routing primitive — direct answer to the episode question

**Task-level position on the spectrum** is implemented by a **router** that chooses among:

1. **In-model routing** (next speaker, next handoff tool, next subagent) — cheap to build, expensive to debug.
2. **Graph routing** (`add_conditional_edges`, `Command`, `Send`) — explicit, testable, still easy to wire wrong (LangGraph static+dynamic edge pitfall).
3. **Application routing** (Pydantic “programmatic hand-off”, your own state machine) — maximum control, you own observability.

What Copilot’s doc-read mode typically **misses** is everything that makes routing real: **handoff transcript filters**, **subagent isolation semantics**, **graph edge composition bugs**, **AutoGen package fork reality**, **τ-bench pass^k**, **leaderboard harness disputes**, and **OTel gen_ai stability flags**. That is the material that lets you defend task-level architecture as engineering rather than as aesthetic preference.
