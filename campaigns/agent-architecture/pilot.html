Agent architecture is a spectrum, not a choice.

And the correct position on it changes per task, not per product. So why does most of the field commit to a position at the product level?

The way the field talks about agent architecture is broken at the level of vocabulary. Monolithic LLM on one side, multi-agent swarm on the other — and the question framed as a product-level commitment. That framing forces a choice the underlying engineering doesn't actually need to make.

What teams pay for, when they commit to one position and stick with it, is a series of mismatches at the task level. A monolith handles tightly-coupled reasoning well and an embarrassingly parallel workload badly. A swarm handles the parallel one well and the coupled one badly. The position on the spectrum is the right abstraction, and the position is task-contingent. The 2025–2026 literature now agrees: relative performance versus single-agent ranges from +80.8% to −70.0%, entirely depending on task type. The architecture that wins on financial reasoning is the architecture that loses on sequential planning by exactly the same amount, in the opposite direction.

The trouble is that the spectrum is not one-dimensional. Number of agents is a vanity axis the field over-uses. The dimensions that decide whether a configuration succeeds or fails are four: how the work is decomposed (the execution unit), what each unit gets to see (the context projection), how much it can spend (the budget), and what makes it stop (the stop policy). Most current frameworks expose only some of these as runtime knobs; many bake at least one in as an architectural assumption.

The switcher below is the menu the spectrum offers. Move the four axes and the topology re-forms. A live verdict describes what the configuration is suited for and what it is hostile to. The exercise is meant to make a quiet point obvious: most of the interesting positions are not at the ends.

Each axis is a control surface frameworks expose to varying degrees. LangGraph exposes all four; CrewAI assumes the execution unit; AutoGen exposes the topology but biases the stop policy; Claude Agent SDK exposes execution and context but defers budget. Most production failures live in the corners where one axis was assumed instead of chosen.

A few configurations earn their keep across most workloads — the chain with projected context and a verifier loop, for instance, is a robust default for sequential reasoning. A few are structurally unstable — an open-ended swarm with full context shared across all agents, for example, has been quietly abandoned by serious teams since 2025. The interactive marks both. What it does not do is tell you which configuration you should pick for your task. That is the routing problem, and it is the genuine open work.

What the field has solved, and what it hasn't

The framework primitives for moving along the spectrum at runtime mostly exist. LangGraph's Command(goto=..., update=...) returns state-and-control-flow in a single object. OpenAI's handoff(input_filter=...) reshapes what the next agent sees. Claude Agent SDK's subagents run in fresh, isolated contexts, with the parent receiving only the final result. Pydantic AI's programmatic hand-off lets application code own the router. These are real, shipped, production primitives. The question of how to move along the spectrum is no longer the bottleneck.

What the field has not built — and what every source consulted for this episode either circles around or names directly — is the router: the mechanism that picks the position per task without requiring a human to configure it for every workload. Six candidate signatures surfaced in the cold round: MasRouter's three-layer cascaded controller (ACL 2025); Gemini-style task taxonomy routers (TRouter, Arch-Router, MoMA); RouteLLM's BERT-class classifier between strong and weak models; ChatGPT's policy over inference graphs (route(task, state, telemetry) → graph); Copilot's four-tuple route(task) → {execution_unit, context_projection, budget, stop_policy}; and the lightweight task-feature classifier the contrarian X discourse keeps reaching for. No two are quite the same. None has yet won.

The convergence under the divergence: the routing primitive is a policy that prices four things together — local difficulty, context entanglement, coordination tax, and debugging liability. The framework primitives enable per-task routing. What is missing is the taxonomy — the human design work that says this kind of task benefits from dispatch, this kind doesn't.

Closing

What you just played with is a menu. What the field still cannot reliably build is the router that picks from it. That is Episode 2's question: what would a working task-feature classifier look like — what features, what training data, what measurement loop? The framework primitives are ready. The taxonomy is not.

The contrarian read that holds the whole thing together comes from Grok's pass through 2026 X discourse: "the correct position changes per task — but building the router that reliably knows is the real unsolved primitive."
