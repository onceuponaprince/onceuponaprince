## 1. PREDICTIVE FEATURES — Which Task Features Survive Contact With Production Traffic?

### Features That Break in Real Workloads

**Ambiguous Intent & Multi-Intent Queries:**
The single most cited production failure is the "hybrid query" problem. Classifiers trained on single-intent benchmarks collapse when users send queries that blend multiple task types. In TRouter (arXiv 2604.09377, April 2026), Liu et al. explicitly flag that existing classification-based routers (RouterDC, MetricRouter) "lack robustness under cross-domain shifts" and fail in **cold-start scenarios** where no in-domain training data exists. This matches Reddit reports from r/LLMDevs where developers note that intent classifiers "fail when encountering new query types (e.g., new product launches or market shifts)" — what the community calls "routing cold starts."

**Model-Reported Confidence:**
arXiv 2602.03478v1 ("When Routing Collapses") documents a pervasive failure mode: routers systematically default to expensive models because they're trained to predict **scalar performance scores** rather than discrete, cost-aware comparisons. Small prediction errors flip relative model rankings, triggering **routing collapse**. This aligns with tianpan.co's production writeup: "Naive cascades rely on model-reported confidence, which is often poorly calibrated. Models may produce fluent, authoritative-sounding, but factually incorrect responses with high confidence."

A Stack Overflow thread (questions/35218922) on kNN embedding classifiers confirms: "A classifier's confidence score is not a reliable proxy for performance; models can be 'correctly' classified with low confidence or 'incorrectly' classified with high confidence (e.g., noise-injected images)."

**Static Heuristics (Regex/Keyword):**
Multiple sources confirm that keyword-based routing is the first casualty in production. arXiv 2505.12601v1 ("Rethinking Predictive Modeling for LLM Routing") notes: "Regex and keyword-based routing fail to capture the intent or complexity of a query, often leading to misrouting when queries are semantically similar but syntactically different." IBM's Spectrum Symphony blog (Jan 2026) reports using KNN with MiniLM embeddings specifically *because* keyword routing was "too brittle."

**Benchmark-Derived Priors:**
arXiv 2604.02367v1 ("Evaluating Small Language Models for Front-Door Routing") explicitly warns: "Models competitive on general benchmarks often diverge on task-specific quality." This is the "benchmark saturation" problem flagged by Galileo.ai and tianpan.co: public benchmarks are "saturated or contaminated, leading to scores that reflect memorization rather than true capability."

### Features That Survive (Barely)

**Task Complexity Scoring:**
The NVIDIA Prompt Task and Complexity Classifier (mentioned in r/LLMDevs thread 1nsi2g7) is one of the few features that survives production contact — but only when **combined with model criteria** (benchmarked performance across task types) rather than used standalone. Developers report shifting from hard-coded model IDs to "model criteria" frameworks.

**Embedding-Based Similarity:**
arXiv 2312.02337v1 ("Measuring Distributional Shifts in Text") finds that clustering-based algorithms on LLM embeddings are "more sensitive to distributional shifts than classical methods." This suggests that embedding-based routers *can* detect drift — but require continuous recalibration.

### GitHub Issue Evidence

**RouteLLM (lm-sys/RouteLLM):**
- **Issue: BERT Router Overfitting** — The original BERT-based routers suffer from "majority class overfitting" with macro F1 scores of 0.23–0.35. The training datasets were heavily skewed (51% strong model wins), and the authors did not release the exact `D_arena` dataset. One community fork (Liqs-v2/RouteLLM) reports preprocessing the claimed 65k samples yielded only **19k usable instances**.
- **Issue: Data Availability** — Users report that without the original dataset, reproduction is impossible. This is the "routing data" problem in action.

**LangGraph (langchain-ai/langgraph):**
- **Issue #6340:** Using a `store` object as a parameter in a conditional edge causes `ValueError: Missing required config key 'store'` at runtime in LangGraph server environments.
- **Issue #4394:** In versions v0.3.32+, `draw_mermaid()` introduced an unexpected, non-functional edge from nodes with conditional branches directly to `END`, misleading developers about actual execution paths.
- **Reddit (r/LangGraph/1rx6dao):** "I think a lot of langgraph debugging goes wrong at the routing step" — the visible failure (wrong node receiving a task) is a symptom of upstream routing logic error.

**Cursor Agent (forum.cursor.com):**

- **Cost drivers:** Labor (60–80%), QA (15–20%), tooling (10–15%).

**Active Learning + LLM-as-Oracle (arXiv 2502.16892v2, Feb 2026):**
Using LLMs to annotate instances selected by active learning "achieves high performance with significantly lower computational and monetary costs compared to direct LLM classification." This is the emerging best practice for solo founders.

**GPT-4-as-a-Judge for Routing Data (Anyscale blog):**
The standard recipe for building RouteLLM training data: generate labeled datasets (query, routing_label) by running responses through both a strong model and a weak model, then using GPT-4 to score which response is better. But this introduces **judge model bias** (see Section 5).

### Synthetic Data Approaches That Failed

**Model Collapse / Autophagy (PMC12778113, arXiv evidence, Reddit r/singularity/1eb7yru):**
- Recursive training on synthetic data causes "model collapse" — models lose the ability to reflect true data distributions, leading to reduced variance, biased analyses, and incoherent outputs.
- **Fidelity Gaps:** Synthetic data "often fails to capture real-world complexities, such as temporal patterns, network hierarchies, or 'messy' human inputs (e.g., speech disfluencies, partial sentences)."
- **Artifactual Relationships:** Generative methods create "artifactual relationships" that distort real-world statistical associations, leading to false correlations.

**In-Vehicle Networks Case (ScienceDirect S2214209625001251):**
Synthetic CAN data "lacks real-world temporal patterns and network hierarchy," showing "reduced identifier coverage and unrealistic temporal patterns" — a concrete example of synthetic data breaking on production features.

**Wireless Signal Case (Pitt Engineering news):**
Current synthetic wireless data has limitations in "affinity" (realism), leading to mislabeled data. "Different tasks have different requirements for data quality."

## 3. CLASSIFIER SHAPE — What Classifier Shapes Have Teams Shipped and Ripped Out?

### LLM-as-Judge

**Shipped:**
- OpenAI Agents SDK handoff filters use LLM-as-judge for routing decisions.
- LangGraph "Router Agent" pattern (dev.to/irubtsov/three-langgraph-agent-patterns) replaces deterministic conditional logic with a dedicated router agent using a `RoutingTool`.

**Ripped Out / Failed:**
- **Reddit r/LLMDevs/1nsi2g7:** "Avoid 'LLM-as-a-Router' — early attempts to use a large LLM to decide routing are often reported as too costly and unreliable."
- **arXiv 2602.03478v1:** "Routing Collapse" — routers systematically default to expensive models. LLM-as-judge trained on performance scores cannot make cost-aware comparisons.
- **Galileo.ai:** "93% of teams report reliability issues due to judge inconsistency, where the same input receives different scores over time."

**Biases:**
- **Position Bias:** Judges favor responses based on order (primacy/recency).
- **Verbosity Bias:** Judges favor longer responses regardless of quality.
- **Self-Enhancement Bias:** Models rate their own outputs higher.
- **Prompt-Induced Bias (arXiv 2604.16790v1):** Judge decisions are "highly sensitive to prompt framing, including authority cues, sentiment, and 'bandwagon' effects."

### Embedding + kNN

**Shipped:**
- IBM Spectrum Symphony (Jan 2026 blog): Uses KNN classifier with MiniLM embeddings for semantic query classification.
- arXiv 2505.12601v1: "When Simple kNN Beats Complex Learned Routers" — kNN is positioned as a "simple yet effective alternative."

**Failed:**
- **Stack Overflow #26939135:** "kNN fails when data is a 'jumble' of classes or lacks clear separability."
- **Outliers:** "Outlier points cause misclassification, as the algorithm will assign a class based on proximity even if the query point is far from the actual cluster."
- **Sensitivity to Scale:** "kNN is highly sensitive to feature scaling; failure to normalize or standardize features often leads to poor performance."
- **Computational Bottlenecks:** "In production, the 'lazy' nature of kNN (searching the entire training space) can be a bottleneck."

### Fine-Tuned Small Model

**Shipped:**
- Anyscale blog: Fine-tuning Llama-3-8B or smaller Qwen variants as routing classifiers.
- Roots.ai blog: Deployed fine-tuned LLMs in production using vLLM framework.

**Failed:**
- **Medium (machine-learning-made-simple):** "Fine-tuning LLMs is a Huge Waste of Time" — "Destructive overwriting: Fine-tuning advanced models risks 'erasing' valuable existing patterns, leading to unexpected downstream effects and loss of general knowledge."
- **Alignment Drift:** "Fine-tuning for safety or alignment can introduce new, unintended biases (e.g., dramatic shifts in demographic distributions)."
- **Inference Throughput:** "Production deployment requires careful management of batch sizes; throughput can drop significantly as input context length increases (e.g., noticeable decline around 8k tokens)."

### Regex / Keyword

**Shipped Everywhere, Ripped Out Everywhere:**
- **arXiv 2505.12601v1:** "Unlike embedding-based routers, regex systems cannot adapt to new, unseen tasks or evolving user language without manual rule updates."
- **Scalability:** "As the number of models or routing categories grows, maintaining a complex set of regex rules becomes an engineering burden."
- **Reddit r/LLMDevs/1nsi2g7:** Teams shifted from keyword matching to "model criteria" (benchmarked performance) specifically because regex was too brittle.

### Hybrid / Multi-Stage / Cascading

**Shipped:**
- **tianpan.co blog:** "Cascading (Waterfall) Routing: Starts with a cheap, fast model and escalates to more capable models only if quality checks fail."
- **TRACER (arXiv 2604.14531v1):** "Parity Gate" — activates a surrogate model only when its agreement with a teacher model exceeds a user-specified threshold.

**Failed:**
- **Miscalibrated Confidence:** "Naive cascades rely on model-reported confidence, which is often poorly calibrated."
- **Latency Accumulation:** "Sequential escalation adds latency at each step. A query requiring the top-tier model may incur the cumulative latency of all preceding tiers."
- **Reddit r/HomeNetworking/556etr:** (Network routing, but analogous) "Double NAT/Firewall Conflicts: Cascading routers often results in double NAT, which can break P2P applications."

**Pattern Convergence:**
- Teams converge on **4-tier classification** (Easy/Medium/Hard/Needs Info) with confidence thresholds at tier boundaries.
- **Morph LLM Router:** Reports ~430ms classification latency, $0.001 per classification, 40–70% cost reduction, <2% quality loss.

## 4. MEASUREMENT LOOP — What Measurement Traps Have Teams Fallen Into?

### Goodhart's Law / Metric Rot

**VentureBeat (infrastructure/monitoring-llm-behavior-drift):**
- "The 'Reliability Gap': Systems may appear fully operational while being 'confidently wrong' because metrics (e.g., high offline pass rates) mask real-world degradation."
- "Evaluation pipelines suffer from 'rot' (concept drift) as user behavior evolves beyond the static 'golden dataset.' Relying on stale data creates a false sense of security where high pass rates mask a degrading user experience."

**Galileo.ai:**
- "Using numeric scales (0-100) introduces more noise than binary (pass/fail) verdicts, which are more consistent and easier to define."

### Distribution Drift Discovery Stories

**orq.ai blog (2025 guide):**
- **Detection Methods:** Population Stability Index (PSI), KL divergence measure deviations from baseline distributions.
- **Clustering-Based Drift (arXiv 2312.02337v1):** Clustering algorithms on LLM embeddings are more sensitive to distributional shifts than classical methods.

**Medium (adnanmasood):**
- **"Agent Drift":** Multi-agent LLM systems suffer "subtle degradation over time" as routing logic becomes stale.
- **Drift-Aware Routing (DAR):** Router logic incorporates "stability scores" to identify drifting agents, allowing for resets or re-routing.

### Leakage Problems

**VentureBeat:**
- **Deterministic Leakage:** "A common production failure is the model generating conversational text when a structured tool call (JSON) is required."
- **Evaluation Leakage:** "When evaluation datasets are not updated with novel production edge cases, the system fails to generalize to new user intents (e.g., a new company policy not in the training/eval set)."

**Cursor Agent (forum.cursor.com):**
- **Context Contamination:** Summaries from unrelated projects injected into new chat sessions, causing "skill leakage."
- **MCP Persistence:** When using MCP-based memory systems, the "active project" state persists across sessions. If a user switches tasks without explicitly resetting, the agent operates in the wrong context.

### Silent Failures

**VentureBeat:**
- "Behavioral telemetry (e.g., users abandoning sessions) often reveals failures that explicit metrics miss."

**LangGraphJS (github.com/langchain-ai/langgraphjs/issues/779):**
- "LangGraph fails silently when exception happens in conditional edge" — the graph stops executing entirely without surfacing an error to the client.

## 5. FAILURE MODES AND FALLBACKS — What Rollback Stories Live in GitHub Issues?

### OpenAI Agents SDK

**Issue #2216 (All handoffs route to last handoff in list):**
- "When multiple agents are created dynamically (e.g., within Temporal workflows), all `handoff()` calls may route to the last agent created in the list, regardless of the intended target. This is attributed to a **closure-related bug**."

**Issue #617 (Handoff Agent Not Triggering Reliably):**
- "Orchestrator agents fail to trigger multiple handoffs as expected, often defaulting to only one agent or failing to execute the handoff entirely."

**Issue #771 (Only 1 handoff getting called no matter what):**
- "Only one handoff agent is being called, and the orchestrator agent is returning the response from that single handoff agent instead of aggregating results."

**Community Pattern (community.openai.com/1235072):**
- "Handoffs can't iterate" — expert agents fail to return control to triage/orchestrator agents, requiring specific prompt engineering.

### RouteLLM

**Issue: LiteLLM Regression (BerriAI/litellm#25629):**
- "Critical regression in `litellm` (v1.82.3) broke OpenRouter model routing when `custom_llm_provider='openrouter'` was set, causing 'not a valid model ID' errors."

**Issue: GPU Deployment Requirement (BerriAI/litellm#4727):**
- "Integrating RouteLLM into proxies like LiteLLM presents architectural concerns, specifically the requirement to deploy on a **GPU** to run the router's classification models."

### Confidence Threshold Patterns

**iblai-openclaw-router (github.com/iblai/iblai-openclaw-router):**
- Uses **weighted scoring** with dimensions: cost, speed, context window, modality support, recency.
- **Thresholds:** `lightMedium`, `mediumHeavy` boundaries tuned using representative workloads.
- **Temporary Disabling:** `systemctl stop iblai-router` while keeping configurations intact.
- **Workload Reversion:** `/config unset models.providers.iblai-router` to revert specific tasks to direct model calls.

**TRACER (arXiv 2604.14531v1):**
- **Parity Gate:** Activates surrogate only when agreement with teacher model exceeds user-specified threshold.
- **Dynamic Calibration:** Refits surrogates on production traces, ensuring threshold calibration remains valid.

**Bifrost Gateway (getmaxim.ai blog):**
- **Circuit Breaking:** Monitors provider health (latency, error rates) and automatically reduces weights or removes failing providers from routing pool.

### When Does the Router Get Yanked Entirely?

**LinkedIn (benvollmer):**
- "LLM-based routing fails in production" due to "constraint violations (e.g., budget, latency, or safety) leading to emergency re-dispatches and missed service-level objectives."

**tianpan.co blog:**
- "Dependency Fragility: Routing systems often fail when a single model provider experiences outages or rate limits. Without multi-provider fallback, the entire routing chain breaks."

**arXiv 2501.01818v1 ("Rerouting LLM Routers"):**
- **Adversarial Robustness:** Routing algorithms can be confounded by "gadget" token sequences designed to force routing to expensive models. "Defenses include filtering high-perplexity queries or monitoring for anomalous user-specific routing frequencies."

## SYNTHESIS: CAN A SOLO FOUNDER BUILD A WORKING TASK-FEATURE CLASSIFIER TODAY?

**Short Answer:** Yes, but it's a 4–6 week grind, and the hard part is **feature engineering + data labeling**, not the model.

### What's Buildable (Framework Primitives)

✅ **RouteLLM framework** exists and works (with caveats about BERT overfitting).  
✅ **Embedding + kNN** is fast and effective for semantic routing (IBM, arXiv 2505.12601v1).  
✅ **LLM-as-judge** for data generation is standard (Anyscale blog).  
✅ **Matrix Factorization** or **Causal LLM Classifier** (Llama-3-8B) are viable model shapes.  
✅ **4-tier routing** (Easy/Medium/Hard/Needs Info) with confidence thresholds is the convergent pattern.  
✅ **Indie hackers** have shipped v0 routers (neuralrouting.io, Frugger, LLmHub.dev).

### What's Not Ready (Routing Data)

❌ **Labeled training data** is scarce. RouteLLM dataset not released; community fork got 19k/65k.  
❌ **Synthetic data** fails due to fidelity gaps, model collapse, artifactual relationships.  
❌ **Cold-start scenarios** break existing routers (TRouter paper explicitly flags this).  
❌ **Distribution drift** requires continuous recalibration; "golden datasets" rot.  
❌ **Feature engineering** is manual: task complexity, semantic similarity, multi-intent queries all require custom classifiers.

### The V0 Recipe (What Solo Founders Are Actually Building)

1. **Proxy-based architecture:** Drop-in replacement for OpenAI SDK (just change `base_url`).
2. **Prompt complexity classifier:** Use NVIDIA's Prompt Task and Complexity Classifier or fine-tune Llama-3-8B on GPT-4-judged data.
3. **Embedding + kNN for semantic routing:** MiniLM embeddings + KNN (k=5–10) for fast classification.
4. **4-tier model selection:** Easy (Haiku/Flash), Medium (Sonnet/Pro), Hard (Opus/Ultra), Needs Info (fallback).
5. **Caching layer:** Cache repeated prompts to avoid redundant API calls.
6. **Confidence thresholds:** Tune boundaries on representative workload (1–2 weeks of prod traffic).
7. **Circuit breaker:** Multi-provider fallback for outages/rate limits.

**Cost:** $0.001/classification, ~430ms latency, 40–70% cost reduction, <2% quality loss (per Morph LLM Router).

**Time:** 4–6 engineer-weeks for v0 (data prep, model training, eval, integration).

**Regret:** If you skip structured logging for routing decisions, you'll be flying blind when drift hits. If you rely on synthetic data without human validation, your classifier will collapse on cold-start queries.

**"A working task-feature classifier is buildable today by a solo founder"** ✅ **TRUE** — Multiple indie hackers have shipped v0 routers in 2025–2026.
