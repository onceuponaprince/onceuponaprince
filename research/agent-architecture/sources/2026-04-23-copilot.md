### 1) Literature (2022–2026): monolithic vs granular performance

**What is stable across papers and benchmarks:**  
1. **Monolithic+tools/scaffold became much stronger** (ReAct-era onward): adding tool use and explicit reasoning loops to one agent often closes much of the gap to multi-agent designs on many tasks.  
2. **Granular/multi-agent helps when decomposition is real**: role-specialisation, planner-executor splits, debate/critique loops, or heterogeneous tool interfaces can outperform one-agent baselines on complex, multi-constraint tasks.  
3. **No universal winner**: results are task-distribution-dependent; coordination overhead and error propagation often erase gains on short or tightly scoped tasks.

**Representative arc:**  
- 2022–2023: ReAct/Toolformer-era shows large gains from *single-agent* scaffold/tool use.  
- 2023–2024: AutoGen/CAMEL/MetaGPT-style work popularises role-specialised multi-agent orchestration.  
- 2024–2026: engineering benchmarks and production reports increasingly show **hybrid routing** wins: start monolithic, escalate to specialist agents only when complexity/uncertainty thresholds are crossed.

### 2) Which frameworks expose the spectrum explicitly vs assume a position?

**Expose explicitly (strong routing primitives):**
- **LangGraph**: graph edges + `Send` + `Command(goto=...)` are first-class routing.
- **OpenAI Agents SDK**: first-class `handoffs`, `Runner` loop semantics, and handoff input filters.
- **AutoGen**: team types (`RoundRobinGroupChat`, `Swarm`) and participant routing as explicit architecture.

**Partly explicit / defaults imply stance:**
- **Claude Agent SDK**: default is one Claude loop, but `agents` subagent config + tool/skill scoping gives controllable granularity.
- **CrewAI**: assumes task-team orchestration as baseline; still lets you collapse to one agent/one task.
- **Pydantic AI**: assumes strongly typed agent units; multi-agent exists but is not the primary orchestration primitive in the core API.

### 3) Where context-stripping improves vs degrades output

**Improves (common):**
1. **Specialist handoff with noisy history removed**: OpenAI example explicitly filters handoff input (`handoff(..., input_filter=...)`) and removes tool chatter/history; this reduces distraction and often improves specialist focus.  
2. **Tool-only workers**: LangGraph `Send("worker", {"subject": ...})` sends only what worker needs.  
3. **Cost-constrained calls**: trimming irrelevant context reduces token latency/cost and can reduce hallucination from stale prior turns.

**Degrades (common):**
1. **Loss of latent constraints**: removed context can drop user preferences, safety constraints, prior commitments.  
2. **Cross-agent inconsistency**: specialists produce locally correct but globally inconsistent outputs when shared state is under-specified.  
3. **Debug opacity**: if stripped context is undocumented, failures become hard to reproduce.

### 4) Coordination-cost curve as agent count scales

**Yes, there is usually a threshold where overhead dominates.**  
Empirically in production systems, cost grows with:
- more turns per task,
- more context serialization/deserialization,
- more handoff failures/retries,
- more observability and replay complexity.

A practical shape is:
- **1–2 agents**: often big gains on complex tasks.
- **3–5 agents**: gains flatten; coordination policy quality dominates.
- **>5 agents**: overhead frequently dominates unless task is naturally parallel and routing is strict.

Framework signals of this cost:
- OpenAI `max_turns` and handoff loop semantics.
- AutoGen team termination/max-turn policies.
- CrewAI process switching (`sequential` vs `hierarchical`) and usage metrics aggregation.
- LangGraph event streams for task-level timing/errors.

### 5) Measurement surfaces that make the spectrum visible

Use these **per task**, not just per product:

1. **Token spend**
   - CrewAI: `calculate_usage_metrics()` aggregates token summaries.
   - Claude SDK types include token/context usage categories.
   - OpenAI/others expose tracing + usage from model providers.

2. **Task completion / quality**
   - Compare monolith route vs granular route on same task class.
   - Track pass@k or rubric scores per route.

3. **Drift rate**
   - How often output violates earlier constraints after handoff.
   - Detect via guardrails/checkers and contradiction scans.

4. **User interventions**
   - Number of manual corrections, retries, forced tool approvals.
   - Claude hooks (`PermissionRequest`, `SubagentStart/Stop`), OpenAI tracing events, LangGraph `"tasks"`/`"debug"` stream modes.

---

### 6) Blind task execution and debugging cost: net reduction or increase?

**Usually net increase in failure cost unless routing and context policy are observable.**

- **Why it can reduce cost:** smaller context windows and narrow specialist prompts can localise errors.
- **Why it often increases cost:** failures become distributed (router, handoff filter, worker prompt, state merge).  
- **Net effect:**  
  - with strong traces/state snapshots (LangGraph checkpoints/tasks, OpenAI trace + session, Claude session/subagent transcripts): often manageable;  
  - without them: debugging cost can exceed the quality gains from granularity.

## Synthesis: task-level routing primitive (answer to Episode 1)

The reusable primitive across frameworks is:

**`route(task) -> {execution_unit, context_projection, budget, stop_policy}`**

- `execution_unit`: one monolithic agent or a specialist/graph node/team.
- `context_projection`: full history, projected state (`Send(arg=...)`), or filtered handoff input.
- `budget`: token/turn/tool limits (`max_turns`, tool allowlists, process constraints).
- `stop_policy`: termination/guardrail conditions.

That is exactly the "spectrum at task level": keep product-level infrastructure constant, but choose routing+context policy per task instance.
